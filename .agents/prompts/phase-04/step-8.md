# Phase 4 Step 8: Create Generate Project Pipeline

## Goal

Create the first simple `generateProject(config)` pipeline in `packages/generator`.

This step should wire together the schema, compatibility validation, feature registry, generation plan, package merge utility, env merge utility, template loader interface, and file tree model.

The goal is a working generator pipeline skeleton, not a complete real template generator yet.

## Scope

### You may

- Add `generateProject(config)`.
- Add `createGenerationPlan(config)`.
- Validate config through `@launchkit/schema`.
- Use compatibility validation from `@launchkit/schema`.
- Resolve enabled features from the feature registry.
- Merge feature `packageJson` contributions.
- Merge feature env var contributions.
- Generate minimal placeholder files from plan data.
- Use an in-memory template loader if one exists.
- Add Vitest tests for the pipeline skeleton.
- Update `progress-tracker.md`.

### You must not

- Add real Next.js templates.
- Add real `shadcn/ui` templates.
- Add real Prisma/Auth.js implementation files.
- Add zip adapter.
- Add filesystem adapter.
- Build website UI.
- Add CLI functionality.
- Install dependencies inside generated projects.

## Required Public API

Add an exported function:

```ts
export async function generateProject(
  config: LaunchKitConfig,
): Promise<GeneratedProject>;
```

If useful, also export:

```ts
export function createGenerationPlan(config: LaunchKitConfig): GenerationPlan;
```

## Required Pipeline Behavior

`generateProject(config)` should:

1. Validate config using `LaunchKitConfigSchema` or a schema helper.
2. Validate compatibility using schema compatibility helpers.
3. Resolve enabled features using `getEnabledFeatures(config)`.
4. Create a generation plan.
5. Merge selected feature `packageJson` contributions.
6. Merge selected feature env var contributions.
7. Create minimal generated files.
8. Return a `GeneratedProject`.

## Minimal Generated Files

Because real templates are not part of this step, generate only a small placeholder project file tree.

Required files:

```txt
package.json
.env.example
README.md
```

Optional placeholder file:

```txt
.launchkit/plan.json
```

Do **not** generate real Next.js app files yet.

That belongs to the template implementation phase.

## `package.json` Requirements

The generated `package.json` should include:

```json
{
  "name": "my-app",
  "private": true,
  "scripts": {},
  "dependencies": {},
  "devDependencies": {}
}
```

It should include merged feature contributions.

For example, Prisma should contribute:

```json
{
  "dependencies": {
    "@prisma/client": "latest"
  },
  "devDependencies": {
    "prisma": "latest"
  },
  "scripts": {
    "db:generate": "prisma generate",
    "db:migrate": "prisma migrate dev"
  }
}
```

Use whatever exact versions were defined in the feature registry.

## `.env.example` Requirements

Use the env merge utility.

If no env vars are required, `.env.example` may be empty or contain a short comment.

If PostgreSQL is selected, include:

```env
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/my_app"
```

If Auth.js credentials is selected, include:

```env
AUTH_SECRET="replace-me"
```

Do **not** generate real secrets.

## `README.md` Requirements

Generate a simple placeholder README with:

- Project name
- Selected features
- Setup commands based on package manager
- A note that this is an early LaunchKit generated skeleton

Example:

```md
# my-app

Generated by LaunchKit.

## Selected Features

- Next.js
- Tailwind CSS

## Getting Started

npm install
npm run dev
```

Use `pnpm` commands when `packageManager` is `"pnpm"`.

## Optional Plan File

If you add `.launchkit/plan.json`, include useful debug information:

```json
{
  "config": {},
  "features": [],
  "env": [],
  "packageJson": {}
}
```

This can help test and preview the pipeline.

If added, it should be clearly marked as temporary or internal.

## Error Handling

If compatibility validation fails, throw a clear error.

If schema validation fails, let Zod errors surface or wrap them in a typed error if the schema package already provides one.

Do **not** swallow errors.

## Testing Requirements

Use **Vitest**.

Do **not** use Node's built-in test runner.

Test at least:

- Default config generates a project.
- Generated project name matches config name.
- Generated project package manager matches config package manager.
- Generated project includes `package.json`.
- Generated project includes `.env.example`.
- Generated project includes `README.md`.
- Generated `package.json` has the correct name.
- Generated README uses npm commands for npm config.
- Generated README uses pnpm commands for pnpm config.
- PostgreSQL config includes `DATABASE_URL`.
- Auth.js credentials config includes `AUTH_SECRET`.
- Prisma config includes Prisma dependencies/scripts.
- Invalid schema config rejects.
- Incompatible config rejects.
- Generated file paths are safe.
